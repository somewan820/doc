# AI 对话事件使用规约（仅使用 text 模板）

本文档是对现有 `事件json规约.md` 的补充。

核心约束：**不新增 JSON 字段，不修改现有事件 schema。**  
大模型对话所需信息统一写入 `outcomes[].text`，由程序按固定标签解析。

---

## 一、使用目标

事件 JSON 继续负责：

- 事件触发
- 属性判定
- NPC 自己的下一事件流转

大模型负责：

- 生成 NPC 本轮开场话
- 生成玩家三个回复
- 根据玩家回复生成 NPC 后续讲述

单轮对话不直接修改主线。主线推进只看事件系统的 `next`。

---

## 二、固定 text 模板

每个 `outcome.text` 必须按以下顺序写入六段：

```text
【上一轮提示词】……
【本轮提示词】……
【对话目标】……
【玩家回复方向】1. …… 2. …… 3. ……
【店铺上下文】……
【本轮记忆摘要】……
```

字段含义：

- `上一轮提示词`：上一轮最重要的状态摘要。第一轮写“无”。
- `本轮提示词`：本轮 NPC 来店状态、购物目的、情绪、可聊与避谈内容。
- `对话目标`：本轮希望玩家感受到的信息或情绪。
- `玩家回复方向`：三个回复选项的语气方向，不写死最终台词。
- `店铺上下文`：货架、时间、天气、客流、价格等经营氛围。
- `本轮记忆摘要`：给下一轮继续携带的一句话摘要。

---

## 三、事件结构约束

继续使用现有字段：

- `version`
- `events`
- `id`
- `type`
- `title`
- `participants`
- `priority`
- `once`
- `trigger`
- `outcomes`
- `next`

第一版经营叙事内容默认只使用 `personal` 事件。

不使用：

- 新增 outcome 字段
- 新增 event 字段
- common 事件串联 NPC
- world 事件推动多 NPC 主线

---

## 四、分线规则

商品不直接决定剧情线。商品只改变 NPC 的统一三维属性：

- `attack`
- `defense`
- `speed`

事件通过 `conditions` 判断属性，决定在关键节点的剧情分支

避免默认死亡线、跨角色联动和复杂多段阈值。

---

## 五、示例 outcome

```json
{
  "id": "AP21E003O001",
  "priority": 100,
  "conditions": [
    { "scope": "npc", "target": "P21", "attr": "attack", "op": ">=", "value": 8 }
  ],
  "text": "【上一轮提示词】江枫上次来店时第一次承认自己没走远，只是在南坊借一口气。【本轮提示词】江枫午后进店，旧剑布条重新缠过，身上的赶路感少了一点。他要买一件攻击法宝，不是为了逞强，而是准备去把一件拖了很久的小事做完。因为攻击属性足够，他语气比前两次稳，愿意承认自己不是被赶着走，而是终于能自己选一段路。【对话目标】让玩家感到江枫的路顺了一些，但仍然只是个人小转身，不是宗门大事件。【玩家回复方向】1. 实用，问他要什么类型的攻击法宝。2. 平静，像对熟客一样说话。3. 轻轻提到他这次看起来不急了。【店铺上下文】午后客人少，货架上有几件普通但顺手的攻击法宝。【本轮记忆摘要】江枫买下攻击法宝，准备去处理自己的旧事，状态比之前更稳。",
  "next": [
    { "target": "P21", "nextId": "AP21E004" }
  ]
}
```

---

## 六、模型输出建议

程序从 `text` 中解析六段后，可要求模型输出：

```json
{
  "npc_opening": "……",
  "player_options": ["……", "……", "……"],
  "npc_followups": ["……", "……", "……"],
  "memory_summary": "……"
}
```

其中 `memory_summary` 可以用于下一轮的 `上一轮提示词`，但事件 JSON 本身仍以策划写入的 `text` 为准。
