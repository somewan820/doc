# 经营叙事 V1 实施说明

本说明对应经营叙事扩展版内容包。

## 一、内容目标

本版剧情从“6 个 NPC 的独立短弧线”升级为：

- 约 `50` 回合的长期经营叙事
- `7` 个核心 NPC 交错出场
- 商品改变 NPC 属性
- 属性在关键节点上决定 NPC 能否完成任务、承受代价或躲开意外
- 中后期 NPC 暂按 `1 个好结局 + 2 个一般结局` 设计
- 前期短线 NPC 可按 `1 个好结局 + 1 个一般结局` 设计
- 剧情树从共同主干逐步分散，最终导向不同结局

玩家体验重点不是单次对话改命，也不是只看人物态度变化，而是通过经营行为改变 NPC 的属性积累，间接决定他们最终会走向哪一种人生。

---

## 二、设计顺序

这一版内容设计必须遵循固定顺序：

1. 先定义 NPC 的固定结局池
2. 再定义导向这些结局的关键属性判定
3. 再倒推剧情分线
4. 最后补充每个事件节点

不要先写过程，再临时拼结局。

## 四、总体结构

### 1. 总回合规划

推荐总流程：

- `R1~R12`：开门立人
- `R13~R30`：主干分叉
- `R31~R42`：分线深化
- `R43~R50`：好结局兑现与结局事件

其中：

- `R1~R48` 作为主要剧情段
- `R49~R50` 作为极少数角色的最终结局事件回合

### 2. 角色安排

- 核心 NPC：`7` 人，承担全部主要剧情树
- 普通散客：只承担经营气氛，不承担固定人生终局

### 3. 单角色标准结构

中后期角色建议包含：

- `2` 个关键判定节点
- `1` 个好结局
- `2` 个一般结局
- `7~9` 个事件节点

前期短线角色建议包含：

- `1` 个关键判定节点
- `1` 个好结局
- `1` 个一般结局
- `5~7` 个事件节点

### 4. 判定规则

- 关键节点不使用“或”逻辑判定。
- 允许单属性门槛，如 `attack >= 13`
- 允许 `&&` 组合门槛，如 `defense >= 8 && speed >= 12`
- 不使用 `||` 组合门槛

### 4.5 隐式联动规则

- 当前阶段允许少量“隐式联动”。
- 隐式联动不是 `common` 多人事件，不让多个 NPC 在同一事件里共同出场结算。
- 写法是在角色 `A` 的关键判定节点里，在某个 `outcome` 分支上额外判断角色 `B` 的属性值。
- 若触发隐式联动，则在角色 `A` 当轮的 `text` 中提及角色 `B` 的痕迹、影响或出手结果。
- 角色 `A` 不必认识角色 `B` 的真实身份，文本可以只保留对玩家可识别、对角色本人未必可识别的暗示。
- 隐式联动主要用于触发隐藏结局，不用于替代该角色自己的主判定逻辑。
- 判定优先级必须写清楚：先判主线好结局，再判隐藏结局，最后才落入普通失败线；或按设计明确写出别的优先级顺序。

### 4.6 关键节点与后续衔接节点的剧情分工

- 关键节点的 `outcomes` 负责写“这一轮判定之后，NPC 当场实际遭遇了什么”。
- 关键节点文本应重点描述：
  - 当场发生的具体事件
  - NPC 是否撑住、失手、受伤、被救、完成任务或错失机会
  - 那一轮最直接的后果
- 关键节点文本不应过早展开：
  - 长期心境总结
  - 回店后的稳定状态
  - 最终去向与后续人生安排
  - 对同一事件的重复复盘
- 若关键节点后接的是一般结局、隐藏结局或好结局前置节点，则后续节点负责写“NPC 经历上一轮事件后，如今回店时的处境、心境、决定和去处”。
- 后续衔接节点应重点描述：
  - NPC 回店时已经进入的状态
  - 玩家从上一轮事件中留下的后果，现在如何体现在 NPC 身上
  - NPC 对未来路线的判断、取舍和去向
- 结局事件尤其要避免重复叙述关键节点里已经写过的“当场发生了什么”。
- 正确分工示例：
  - 关键节点写“白芷在最险一击前被断剑剑客搭救”
  - 后续结局事件写“白芷惊魂稍定后，决定暂离南坊，回药王谷磨练自身，暂不再独自承担高难度事务”
- 好结局前置节点同理：
  - 关键节点写“她当场撑住了最高压考验”
  - 前置节点写“她事后已开始像一个能接住事情的人那样思考和补货”

### 5. 结局事件规则

- 每个结局都必须有明确的结局事件。
- 结局事件必须由 NPC 本人亲自回店触发，不使用口信、留物、旁人转述等间接方式。
- 店主在结局事件中的主要功能是倾听者和见证者。
- NPC 可以在结局事件中报喜或报忧，说明接下来的动向、离去或留下。
- 结局事件可以保留角色不愿暴露的隐秘目的，不要求把一切真相说透。
- 结局事件使用 `E` 开头，但整体结构保留原有样式，即把原本前缀中的 `A` 替换成 `E`，例如 `AP22E009 -> EP22E009`。

## 六、阶段与商品档位

| 阶段 | 核心角色 | 商品数值参考 | 叙事功能 |
| ---- | -------- | ------------ | -------- |
| 前期 | 崔远、费仁 | `Common`：单项约攻 0~3、防 0~2、速 0~4 | 让玩家理解“低端货也会改命” |
| 中期 | 江枫、白芷、陈洁 | `Fine`：单项约攻 0~10、防 0~8、速 0~15 | 让剧情从谋生感进入选择感 |

## 七、剧情树结构

不再采用“每个 NPC 固定 4 轮”的模板，也不采用“简单 A/B 分线”的模板。

中后期角色建议统一改为：

1. 共同开场节点
2. 主干推进节点
3. 第一次关键判定
4. 第一个一般结局快速收尾节点
5. 主干继续推进节点
6. 第二次关键判定
7. 第二个一般结局快速收尾节点
8. 好结局前置节点
9. 好结局兑现节点

前期短线角色建议统一改为：

1. 共同开场节点
2. 主干推进节点
3. 关键判定节点
4. 一般结局快速收尾节点
5. 好结局前置节点
6. 好结局兑现节点

### 关键原则

- 关键判定影响的是“这个人接下来的人生能走到哪里”，而不是一句话心情变了。
- 好结局应尽量走完整条主干，并推迟到合理回合兑现。
- 一般结局应尽量在中段触发，并在 `1` 轮内收尾退场。
- 前期角色的好结局不必拖到终盘，可以在游戏中期完成收尾。
- 无论好坏结局，最终都应安排一次 NPC 本人回店的结局事件。
- 隐藏结局若存在，也应有明确优先级和明确人生后果，不能写成模糊彩蛋。
- 每个结局都要能倒推出一条清晰的前置路径。
- 如果某条分支无法明确通向哪个固定结局，就说明它不该存在。

## 八、事件 ID 建议

现有实现中，普通过程事件沿用角色线编号；结局事件改为 `EPxxExxx` 这种样式。

| NPC | 建议事件范围 |
| --- | ----------- |
| 江枫 P21 | 过程事件 `AP21E001` ~ `AP21E008`；结局事件 `EP21E004` / `EP21E007` / `EP21E009` |
| 白芷 P22 | 过程事件 `AP22E001` ~ `AP22E008`；结局事件 `EP22E004` / `EP22E007` / `EP22E009` |
| 殷无邪 P23 | 过程事件 `AP23E001` ~ `AP23E008`；结局事件 `EP23E004` / `EP23E007` / `EP23E009` |
| 崔远 P24 | 过程事件 `AP24E001` ~ `AP24E006`；结局事件 `EP24E004` / `EP24E007` |
| 沈七 P25 | 过程事件 `AP25E001` ~ `AP25E008`；结局事件 `EP25E004` / `EP25E007` / `EP25E009` |
| 费仁 P26 | 过程事件 `AP26E001` ~ `AP26E006`；结局事件 `EP26E004` / `EP26E007` |
| 陈洁 P27 | 过程事件 `AP27E001` ~ `AP27E008`；结局事件 `EP27E004` / `EP27E007` / `EP27E009` |

建议约定：

- `E001~E003`：共同主干与第一次关键判定
- `E004`：第一次一般结局快速收尾
- `E005~E006`：主干继续推进与第二次关键判定
- `E007`：第二次一般结局快速收尾
- `E008`：好结局前置
- `EPxxExxx`：结局事件本身

前期短线角色可以压缩使用同一套编号位，不必强行占满复杂结构。

不在普通过程事件 ID 上硬编码 `A/B`，而是在内容设计上明确树形导向。

## 十、text 模板

所有 `outcome.text` 仍使用固定标签：

```text
【上一轮提示词】……
【本轮提示词】……
【对话目标】……
【玩家回复方向】1. …… 2. …… 3. ……
【店铺上下文】……
【本轮记忆摘要】……
```

扩展建议：

- 对关键节点补充“本轮若失败，会失去什么”
- 对终局节点补充“这次来店是否已经是此人的结局事件”
- 对关键节点与后续衔接节点严格分工：前者写当场遭遇，后者写事后处境与未来去向

## 十一、实现注意

- 商品只改变属性，不直接写死结局。
- 结局由“属性累计结果 + 关键节点判定 + 分支路径”共同决定。
- NPC 最终判定属性 = NPC 基础属性 + 玩家卖给该 NPC 的商品属性累计。
- 商品 `movementSpeed` 需要累加到 NPC 的 `speed`。
- 不新增 JSON 底层字段的前提下，可以通过更多事件节点承载树状分线。
- 当前阶段先不要写 `common` 事件，优先把每个 NPC 的个人线理顺。
- 隐式联动可以写，但必须保持“个人线为主、他人属性为辅”的结构。
- 不要把事件写成纯情绪变化，应尽量对应具体人生后果。
- 若一个关键节点后面还有结局事件或前置事件，则不要在关键节点和后续节点里反复讲同一件事；应把“事发当场”和“事后收束”拆开写。

推荐落地顺序：

1. 先写所有 NPC 的固定结局
2. 再写关键节点的必要条件
3. 再写哪一个节点触发哪一个一般结局
4. 最后写主干和好结局事件文本
